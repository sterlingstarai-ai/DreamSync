## 🎯 Group B 핵심 토론 4가지

### **토론 1: 온보딩 최적화 - Dashboard 카드 축소 vs 샘플 데이터 자동 로드**

#### 현재 상황
- **UX Research 발견**: Dashboard 정보 과다 (7개 카드) → 신규 사용자 혼란
  - Forecast + CoachPlan + GoalRecovery + ForecastReview + QuickActions + Stats + RecentDreams
  - 첫 분석까지 대기 시간 → 체크인 임무감 저하
  - 드롭오프 예측: 30-40% (평균 50-60% 대비 양호하나 개선 필요)

- **Growth 제안**: 온보딩 3단계에서 **샘플 데이터 자동 로드**
  - "첫 꿈 기록 없이도 앱 가치를 즉시 체험"하게 함
  - Dashboard 채워진 상태로 진입 → 심리적 불안감 해소

- **CRM 분석**: L0(신규) → L1(탐색) 전환이 가장 중요한 병목
  - 온보딩 완료 후 첫 3일 내 "꿈 기록 1개 + 체크인 1회" 달성 여부가 L1 판별 기준

#### 토론 요점

**의견 A (UX Research)**: "Dashboard 카드 축소가 먼저"
```
근거:
1. 신규 사용자에게는 "꿈 기록" + "체크인" 2개만 필요
2. QuickActions, Stats, RecentDreams는 L2+ 단계에서만 의미 있음
3. CoachPlan, GoalRecovery는 데이터 부족 상태에서 noise만 늘림
4. 명확한 다음 단계("내일 저녁 체크인 완료하기")만 표시하면 인지 부하 50% 감소

구현:
- L0-L1: Quick Actions(꿈, 체크인) + Next Step Guide만 표시
- L2+: 필요한 카드만 Progressive Disclosure
```

**의견 B (Growth)**: "샘플 데이터가 먼저"
```
근거:
1. 온보딩 중 샘플 꿈 데이터 자동 로드 (1-2개)
2. AI 분석 결과 미리보기 (심볼, 감정, 테마)
3. 예보 카드 자동 생성 ("오늘의 컨디션 예측")
4. Dashboard 채워진 상태 = 앱 가치 즉시 증명 = 동기 부여

데이터:
- Baseline: D1 내 꿈 기록 60% → 목표 80%
- 샘플 데이터 로드 후: 70-75% 기대
```

**의견 C (CRM)**: "두 가지 모두 필요, 단계적 적용"
```
추천 순서:
Phase 1 (주 1-2):
  - Dashboard 카드 축소 (신규 사용자 혼란 해소 = 빠른 효과)
  - 실행: UI 조건부 렌더링만 필요 (backend 불필요)
  - 효과: D3 리텐션 +5-10pp 즉시

Phase 2 (주 3-4):
  - 샘플 데이터 자동 로드 추가 (동기 부여 심화)
  - 실행: Onboarding 3단계에서 API 호출 (Backend: 샘플 꿈 데이터 엔드포인트)
  - 효과: D1 꿈 기록율 60% → 75%, D3 리텐션 추가 +3-5pp
```

#### **최종 결론 (Group B 권고)**
```
✅ 순서: Dashboard 축소 FIRST, 그 후 샘플 데이터 로드

근거:
1. Dashboard 축소는 "인지 부하 감소" (확실한 효과, 위험도 낮음)
2. 샘플 데이터는 "동기 부여" (효과 좋으나 데이터 복잡도 증가)
3. 먼저 혼란을 제거해야 동기 부여가 효과적

구현 전략:
- 주차 1: Dashboard L0-L1 별도 렌더링 (UI/UX)
- 주차 2-3: UX Research 신규 사용자 5명 검증
- 주차 3-4: 샘플 데이터 엔드포인트 준비 (Backend)
- 주차 4-5: 통합 A/B 테스트 (Growth 주도)
```

### **토론 2: 리텐션 강화 vs 바이럴 구축 - 어디에 투자할 것인가?**

#### 현재 상황
- **CRM 평가**: 리텐션 루프 9/10 (매우 우수)
  - 일일 루프 명확: 아침(꿈) → 낮(예보) → 저녁(체크인) → 주간(리포트)
  - 알림 + 코치 플랜 + 패턴 감지 시스템 완비
  - **다만 재참여 전략 전무**: L4(이탈 위기) → L2(활성)로의 명시적 플로우 없음

- **Growth 분석**: 바이럴 K≈0
  - K = i × c (i: 초대 인센티브 비율, c: 초대 수락율)
  - 현재 feature: 공유 버튼 없음, 소셜 초대 기능 미구현
  - **이유**: 꿈과 감정 = 민감한 개인정보 → 소셜 공유 어려움

- **PMM 전략**: "예측형 셀프케어" 포지셔닝
  - TAM $400억+ (수면 + 멘탈헬스 + 웰니스)
  - 경쟁 차별화: 컨디션 예보 + AI 코치 (유일)
  - 시장 진입 전략: 유기 성장 (viral보다는 수렴하는 호평)

#### 토론 요점

**의견 A (Growth)**: "바이럴 구축 필수"
```
근거:
1. CAC(고객획득비용) 감소 = 비즈니스 sustainability
2. 현재 광고/PR 채널 없으면 User Acquisition 불가능
3. 소셜 공유 기능 추가 = 리스크 낮음 (선택사항)

제안:
- 심볼 사전 공유: "내 꿈에 나타난 [심볼]이 뭘까?" (유일한 공유 포인트)
- 주간 리포트 스냅샷: "내 이번 주 웰니스 지수는 [점수]"
- 친구 초대: "함께 꿈을 분석하는 친구 있나요?" (선택적)

목표: K = 0 → 0.2 (2개월 내)
```

**의견 B (CRM)**: "바이럴 투자는 리텐션 부작용"
```
근거:
1. 리텐션 9/10은 이미 매우 높음 (경쟁사 대비 2배)
2. 바이럴 추구 = 사용자 행동 변화 강요 → 유기적 리텐션 저해
3. 예: "심볼 공유" 압박 → 민감한 꿈 공개 거부 → 불만

현실:
- DreamSync는 "혼자만의 성장" 앱 (일기장 같음)
- Sleep Cycle, Calm도 소셜 공유 비중 낮음
- 초기: 리텐션 극대화 (MAU 10K 달성) → 그 후 바이럴 고려

시장 데이터:
- 웰니스 앱 평균 바이럴 K = 0.1 이하
- DreamSync는 K=0으로 시작해도 리텐션만으로 성장 가능
```

**의견 C (UX Research + PMM)**: "하이브리드 전략: 리텐션 우선, 바이럴 선택적"
```
조율안:
Phase 1 (즉시 - 2개월):
  - 리텐션 집중: 생애주기 알림, 온보딩 최적화
  - 측정: D30 리텐션 15% → 20%+ 달성
  - 바이럴은 보류

Phase 2 (2-4개월):
  - 심볼 공유만 추가 (비강제적, 미적)
  - A/B 테스트: 공유 기능 있음 vs 없음
  - 목표: K = 0 → 0.05-0.1 (기대 이상이면 충분)

Phase 3 (4개월+):
  - 데이터 기반 판단
  - D30 리텐션 20%+ 달성했으면 → 바이럴 투자
  - 달성 못했으면 → 리텐션 심화
```

#### **최종 결론 (Group B 권고)**
```
✅ 리텐션 우선 전략 (K=0 수용)

근거:
1. DreamSync는 "혼자의 성장"을 가치로 함
2. 현재 리텐션 9/10은 이미 매우 높음
3. 바이럴 추구하면 핵심 가치(프라이버시)를 훼손할 수 있음

실행 로드맵:
- 즉시: 생애주기 알림 + 온보딩 최적화 (Group B 권고 #1, #3)
- 2개월: D30 리텐션 15% → 20%+ 달성 확인
- 4개월: 심볼 공유 시범(선택적, 비강제)
- 그 이후: 데이터 기반 추가 바이럴 기능 검토

투자 배분:
- 리텐션: 70% (알림, 온보딩, 코치 플랜)
- 바이럴: 0% (현재), → 10% (4개월 후 재검토)
- 신규 기능: 20% (UHS, 웨어러블, 코칭)
```

### **토론 3: 알림 개인화 - 3종→20개 시나리오, 기술적 구현 가능성**

#### 현재 상황
- **현재 알림**: 3종 (아침 꿈 07:00, 저녁 체크인 21:00, 주간 리포트 일요일 10:00)
  - 모든 사용자에 동일
  - 고정 문구 ("어젯밤 꿈을 기억하시나요?")
  - 메시지 개인화 0%

- **CRM 제안**: 20개 자동화 시나리오
  - 9단계 생애주기 (L0~L5) × 스트레스/컨디션/수면 세그먼트
  - 각 단계별 맞춤 메시지 + 최적 발송 시간
  - 예: L4 사용자 3일 무활동 → "2일 안 뵙는데..." 복귀 알림

- **기술 복잡도**: 높음 (useNotifications.js + 서버 로직 필요)
  - 현재: Capacitor LocalNotifications 기반 (로컬 알림만)
  - 확장: 원격 푸시 + 동적 메시지 + A/B 테스트 + 영향 측정

#### 토론 요점

**의견 A (CRM)**: "20개 시나리오 모두 구현해야 함"
```
근거:
1. 리텐션 루프 9/10 = 기초는 견고
2. 재참여 전략 전무 = 가장 큰 gap
3. L4→L2 복귀율 50% 향상 가능 (현재 미측정 → 목표 30%)

기술 실행:
- useLifecycleStore: 사용자 생애주기 단계 추적
- messageBuilder: 단계별 동적 메시지 생성
- notificationScheduler: 최적 시간 계산 + 스케줄

구현 난이도: 높으나 "핵심 경쟁력" 수준

예시 (3개 시나리오):
L0 (신규) → "첫 꿈을 기록해볼까요?" (즉시)
L4 (위기) → "2일 안 뵙는데, 어느 신세요?" (3일 무활동 시)
L3 (습관) → "월요일은 스트레스가 높으신데, 산책은?" (패턴 기반)
```

**의견 B (UX Research)**: "단계적 구현: MVP(5개) → 확장(20개)"
```
근거:
1. 20개 모두 동시 구현 = 버그 + 성능 문제 + 테스트 어려움
2. 한번에 너무 많은 변수 = 어느 알림이 효과적인지 불명확

MVP 단계 (주 1-2):
  - L0: "첫 꿈 기록 유도" (1개)
  - L1: "체크인 습관화" (매일 저녁, 1개)
  - L4: "복귀 알림" (3일 무활동 시, 1개)
  - L3: "주간 인사이트" (주 1회, 1개)
  - 추가: "마일스톤" (3일 연속 기록, 1개)

Effect Size 측정 후 확장:
  - D3 리텐션 개선도?
  - 알림 CTR?
  - 메시지별 전환율?

그 후 추가 시나리오:
  - 스트레스 고위험 (L2-L3)
  - 컨디션 저하 (L2-L3)
  - 수면 부족 (모든 단계)
  - 악몽 패턴 (L1-L3)
```

**의견 C (Growth)**: "A/B 테스트 능력이 먼저"
```
현실 문제:
1. 현재 Analytics 연동 전무 → "알림 CTR 측정 불가"
2. "20개 시나리오" = 20개 가설 = 검증 능력 필요

제안:
Phase 1: Analytics + 기본 알림 메시지 A/B 테스트 기반 마련
  - "어젯밤 꿈을 기억하시나요?" vs "어젯밤 어떤 꿈을 꾸셨어요?"
  - 어느 메시지가 꿈 기록율을 더 높이는가?

Phase 2: 단계별 알림 추가 (A/B 능력 있을 때)
  - L0 환영 알림
  - L4 복귀 알림
  - 각 효과 측정

Phase 3: 확장
  - 데이터 기반으로 어떤 알림이 효과적인지 파악 후 추가
```

#### **최종 결론 (Group B 권고)**
```
✅ 단계적 구현: MVP 5개 (주 2-4) → 확장 20개 (주 5+)

MVP 알림 (우선순위):
1. L0 신규: "첫 꿈 기록" (온보딩 후 즉시)
2. L1 탐색: "오늘 저녁 체크인" (매일 21:00, 메시지는 동적)
3. L4 위기: "3일 무활동 복귀" (L4 진입 후 1회)
4. L1 마일스톤: "3일 연속 기록 축하" (조건부, 심리 강화)
5. L3 습관: "주간 인사이트" (일요일 10:00, 패턴 기반)

기술 준비 (주 1):
- useLifecycleStore: L0~L5 단계 추적 (CRM)
- messageBuilder: 동적 메시지 (Frontend)
- Analytics: 알림 CTR 측정 (Growth, Backend)

효과 측정 (주 2-4):
- D3 리텐션: 40% → 50% (목표)
- 알림 CTR: 기준선 수립
- 메시지별 전환율: 어느 알림이 "꿈 기록" 유도에 효과적?

확장 (주 5+):
- 데이터 기반으로 20개 시나리오 추가
- 스트레스/컨디션/수면 세그먼트 기반 메시지
```

### **토론 4: WCAG 접근성 - 명암비 개선이 리텐션에 미치는 영향**

#### 현재 상황
- **UI/UX 발견**: 명암비 WCAG AA 미달
  - text-secondary (#a0a0b0) + bg-tertiary (#252540) = 3.8:1 (목표: 4.5:1)
  - 주로 보조 텍스트 영향 (기본 텍스트는 16:1로 우수)
  - Focus-visible 스타일 미지정 (키보드 사용자 경험 저하)

- **UI/UX 효과 예측**: "명암비 개선 = 접근성 개선 = 리텐션 미미"
  - 대다수 사용자는 시각장애 없음
  - 다크 모드는 이미 OLED 친화적
  - 색맹 사용자: 한국 인구 8-10% (약 400만명)

#### 토론 요점

**의견 A (UX Research)**: "접근성도 중요하지만, 리텐션 영향은 미미"
```
근거:
1. 명암비 개선의 주요 수혜자: 색맹 사용자 8% + 노령층 + 저시력
2. DreamSync 1차 타겟 (25-35세 "의식적 자기 탐구자")와 직접 관련도 낮음
3. 개선 작업: CSS 변수 1-2개 조정만으로 충분 (높은 난이도 아님)

리텐션 영향:
- 직접 영향: ~1-2pp (색맹 + 접근성 니즈 충족)
- 간접 영향: 0 (타겟 페르소나 구성상)

권장: "하지만 해야 한다" (규정 + 포용성)
```

**의견 B (CRM)**: "접근성 = 장기 신뢰도 = 간접적 리텐션"
```
근거:
1. "접근성 신경쓰는 앱" = 신뢰도 높은 브랜드
2. 웰니스/멘탈헬스 앱은 "안전함 & 신뢰" = 핵심 가치
3. 접근성 공시 + WCAG AA 준수 = 마케팅 포인트

리텐션 영향:
- 직접: 접근성 필요 사용자 ~ 1-2pp
- 간접: "배려하는 브랜드" 신뢰도 → 구전 + 재구매 의향 +5pp
- Focus-visible 추가 = 키보드 사용자 경험 향상 → 포용성 개선

제안: "투자 가치 있음" (작업량 대비)
```

**의견 C (PMM + Growth)**: "빠른 출시 vs 완벽성 트레이드오프"
```
현실 체크:
1. 시장 경쟁: Elsewhere, Sleep Cycle 이미 출시 (점유 경쟁)
2. 출시 일정: 3-4주 남음
3. WCAG AA 준수: 2-3일 작업 (색상 조정 + focus-visible)

전략:
- MVP 출시: 현재 상태 (WCAG AA 미달)
- 1개월 후: 접근성 개선 패치
  * 명암비 조정: text-secondary #b5b5c5 (30분)
  * Focus-visible 추가: 1시간
  * 색맹 시뮬레이션 테스트: 1시간

비용-효과:
- 연기 비용: 시장 점유 손실 >> 접근성 개선 이득
- 권장: 출시 후 즉시 Patch 1.0.1 (1주 후)
```

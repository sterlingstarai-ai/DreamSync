## 1. 사용자 여정 맵

### 1.1 전체 여정 플로우

```
신규 사용자
    ↓
[Auth] 로그인/회원가입 (게스트 옵션)
    ↓
[Onboarding] 3단계 안내 + 알림 권유 + 샘플 데이터
    ↓
[Dashboard] 홈 진입 (아침 인사말, 빈 상태)
    ↓
┌─────────────────────────────────────┐
│  일일 루틴 (Daily Core Loop)         │
├─────────────────────────────────────┤
│ 1️⃣ 아침: DreamCapture (꿈 기록)      │
│    - 텍스트 입력 + 음성 녹음          │
│    - AI 분석 (심볼, 감정, 테마)       │
│                                     │
│ 2️⃣ 오후: Dashboard 예보 확인 (자동)  │
│    - 신뢰도 + 추천 행동 + 패턴 알림   │
│                                     │
│ 3️⃣ 저녁: CheckIn (30초 체크인)       │
│    - 컨디션(1-5) → 감정(최대5개)      │
│    - 스트레스(1-5) → 이벤트 다중선택  │
│    - 수면 정보(healthkit 옵션)        │
│                                     │
│ 4️⃣ 주간: WeeklyReport (분석)         │
│    - 차트, 감정 통계, 심볼 빈도      │
│    - 목표 진행도, 코치 플랜 제안      │
└─────────────────────────────────────┘
    ↓
[Secondary] 탐색 페이지
    - SymbolDictionary: 심볼 검색 + 의미 학습
    - TimelineSearch: 꿈 기록 검색 + 필터
    - Settings: 알림, 개인정보, Feature Flags
    ↓
[Retention Paths]
- 연속 기록 스트릭 유지
- 예보 정확도 추적
- 목표 달성 코치 플랜
- 웰니스 지수(UHS) 모니터링
```

### 1.2 사용자 단계별 감정 곡선

| 단계 | 감정 상태 | 주요 니즈 | 리스크 |
|------|---------|---------|-------|
| **회원가입** | 호기심 + 약간의 불안 | 빠른 진입, 신뢰도 | 복잡한 가입 프로세스 |
| **온보딩** | 설렘 + 교육욕 | 가치 이해, 첫 성공 | 단계 건너뛰기 |
| **첫 기록** (Dream) | 만족감 + 기대감 | 쉬운 입력, 즉시 결과 | 시간 소요, AI 오류 |
| **체크인** | 루틴화 (긍정) | 빠르고 간단함 | 번거로움, 잊음 |
| **리포트 확인** | 자기인식 + 뿌듯함 | 의미 있는 인사이트 | 데이터 부족 느낌 |
| **1주일 이상** | 습관화 + 몰입 | 지속적 개선, 커뮤니티 | 정체기, 이탈 위험 |

## 2. 온보딩 플로우

### 2.1 구조 분석 (Onboarding.jsx)

**단계 수**: 3단계 (보통 범위)

| 단계 | 콘텐츠 | 액션 | 마찰도 | 비고 |
|------|--------|------|-------|------|
| **Step 1: Welcome** | 환영 메시지 + 가치 제안 | "다음" 또는 "건너뛰기" | 낮음 | 아이콘 애니메이션, 명확한 가치 |
| **Step 2: How-To** | 핵심 루프 설명 | "다음" 또는 "건너뛰기" | 낮음 | 3단계 요약 (기록→체크→예보) |
| **Step 3: Permission** | 알림 권유 + 샘플 데이터 | 알림 허용 / 샘플 로드 / 건너뛰기 | 중간 | 선택 사항이지만 권유 강함 |

### 2.2 긍정적 요소

✅ **간결성**: 3단계는 이상적 범위
✅ **스킵 가능**: 모든 단계에서 "건너뛰기" 제공
✅ **시각적 진행**: 프로그레스 도트로 현재 위치 표시
✅ **명확한 가치 제안**: "더 나은 하루를 예측해드려요"
✅ **첫 성공 경험**: 샘플 데이터로 즉시 앱 가치 체험
✅ **토너리스/톤앤매너**: 친근하고 격려적 ("환영해요", "도와드려요")

### 2.3 마찰 포인트 & 드롭오프 위험

⚠️ **알림 권유 모호성**
- `handleEnableNotification()`은 성공/실패 경로가 다르게 처리됨
- 실패 시 `updateNotifications({ enabled: false })` 호출 (사용자가 거절한 의도 반영 X)
- **개선**: "나중에"와 "거절" 옵션을 명시적으로 분리

⚠️ **샘플 데이터 로드 상태 모호성**
- `sampleStatus` = 'idle' | 'added' | 'existing'
- 사용자가 이미 샘플을 로드했는지 확인 불가 (새로 들어올 때마다 시도 가능)
- **개선**: 로드 후 버튼 비활성화 또는 명확한 상태 피드백

⚠️ **데이터 이전 없음**
- 온보딩 후 직 대시보드로 이동 (중간 트랜지션 상태 X)
- 첫 기록 전까지 빈 화면 → 초기 불안감

### 2.4 드롭오프 예측 시나리오

| 시나리오 | 확률 | 예상 경각 |
|---------|------|---------|
| 알림 거절 후 즉시 앱 종료 | 중간 | ~20% |
| 온보딩 스킵 (건너뛰기) → 대시보드 혼란 | 중간 | ~15% |
| 샘플 데이터 없이 빈 대시보드 진입 | 낮음 | ~5% (UI 명확) |
| iOS 앱 설치 후 권한 거절 | 중간 | ~25% (OS 수준) |

**총 드롭오프 위험**: ~30-40% (평균 앱 온보딩 이탈률 50-60% 대비 양호)

## 3. 핵심 사용 루프

### 3.1 일일 루틴 분석

#### 3.1.1 아침: Dream Capture (꿈 기록)

```
DreamCapture.jsx 분석
├─ 입력 폼
│  ├─ Textarea: 자동 높이, 6줄 기본
│  ├─ 음성 녹음: Web Speech API (지원 시)
│  └─ 저장 버튼: 검증 (10자 이상 필수)
├─ AI 분석 결과
│  ├─ 요약 (1-2문)
│  ├─ 강도 (1-10 + 감정 레이블)
│  ├─ 심볼 (다중 발견)
│  ├─ 감정 (색상 코딩)
│  ├─ 테마 (다중)
│  └─ 자기성찰 질문 (5-6개)
└─ 최근 기록 목록 (최대 10개 표시)
```

**UX 강점**:
- 음성 입력 옵션으로 텍스트 입력 피로 감소
- AI 결과는 호기심 유발 (즉시 의미 발견)
- 반복 기록 용이 (최근 목록 항상 보임)

**마찰점**:
- 10자 최소 요구는 합리적이나, "조금 더 자세히" 토스트가 모호함
- 분석 로딩 중 "분석 중..."만 표시 → 체감 시간 길게 느낄 수 있음
- 결과 닫기 후 입력 폼으로 자동 스크롤 안 함 → 두 번째 기록 시 불편

#### 3.1.2 오후: Dashboard 예보 확인

```
Dashboard.jsx 분석 (자동 생성, 최소 데이터 필요)
├─ Forecast Card
│  ├─ 조건 레이블 + 신뢰도 %
│  ├─ 요약 (1-2문)
│  ├─ 추천 행동 (최대 3-4개)
│  │  └─ 토글 가능 (실천 체크)
│  └─ 면책 문구 ("참고 지표")
├─ Pattern Alert Card
│  ├─ 감지된 패턴 (심각도별 색상)
│  ├─ 설명 + 추가 경보 개수
│  └─ 자동 표시 (조건: 3+ 최근 로그)
├─ Coach Plan Card (목표 기반)
│  ├─ 오늘 태스크 (최대 5개)
│  ├─ 진행률 바
│  └─ 토글 완료 (클릭)
├─ Goal Recovery Card
│  ├─ 주간 목표 대비 부족분
│  ├─ 오늘 플랜에 추가 (병합)
│  └─ 남은 일수 표시
└─ Quick Actions
   ├─ 꿈 기록 (오늘 개수 or "아직 기록 없음")
   └─ 체크인 (완료 여부)
```

**UX 강점**:
- 한눈에 오늘 상태 파악 (다층 정보 구조화)
- 추천 행동 실시간 체크 → 즉시 피드백
- Quick Action으로 다음 스텝 명확
- 예보 없는 상태도 친절한 메시지 ("첫 예보를 기다리고 있어요")

**마찰점**:
- 정보 과다 (7-8개 카드) → 스크롤 필요 = 인지 부하
- "신뢰도 %" 설명 없음 → 의문점 (왜 70%? 근거?)
- Coach Plan + Goal Recovery 동시 표시 시 중복 느낌
- 예보 리뷰 ("어제 예보 검증") 매일 나타남 → 피로도

#### 3.1.3 저녁: CheckIn (30초 체크인)

```
CheckIn.jsx 분석 (멀티 스텝 폼)
Step 1: Condition (1-5 이모지)
Step 2: Emotion (최대 5개 선택)
Step 3: Stress (1-5 슬라이더)
Step 4: Sleep (healthkit 플래그 on일 때 - 취침/기상/품질)
Step 5: Events (다중 선택, 카테고리별)
└─ 진행률 바 (시각적 진행)
```

**목표**: 30초 완료 (실제 시간 측정 필요)

**현재 실제 시간**:
- Step 1 (Condition): 3초 (이모지 선택)
- Step 2 (Emotion): 10-15초 (최대 5개, 고민)
- Step 3 (Stress): 3초 (슬라이더)
- Step 4 (Sleep): 20-30초 (시간 입력, 품질 선택)
- Step 5 (Events): 10-20초 (다중 선택, 스크롤)
- **예상 합계**: 50-80초 (목표 초과)

**UX 강점**:
- 한 화면 하나 질문 (인지 부하 최소)
- 이모지/칩 기반 빠른 선택
- 이전 버튼으로 수정 가능
- 진행률 바로 동기부여

**마찰점**:
- **실제 30초 목표 달성 불가** (특히 감정/이벤트 단계)
- healthkit 데이터 자동 채움되어도 시간 입력 필요 (신뢰도 확인?)
- 이벤트 카테고리별 스크롤 필요 → 모바일에서 번거로움
- 완료 후 대시보드로 자동 이동 → 결과 확인 불가

#### 3.1.4 주간: WeeklyReport (분석)

```
WeeklyReport.jsx 분석
├─ 일별 컨디션 스파크라인
├─ 감정 top 5 (빈도)
├─ 심볼 top 10 (최근 일주일)
├─ 수면/스트레스 추이
├─ 목표 진행도 (주간 목표별)
├─ 코치 플랜 통계 (완료율)
├─ 예보 정확도 (hit/miss)
└─ 추천 목표 제시 (AI 기반)
```

**UX 강점**:
- 종합적 통찰력 제공 (자기인식 도구)
- 숫자 + 시각화로 설득력 높음
- 목표 기반 피드백

**마찰점**:
- 정보 과다 (10+ 섹션) → 한눈에 이해 어려움
- 목표 추천 후 적용 UX 불명확
- 모바일 화면에서 차트 가독성 문제 (작은 화면 + 많은 데이터)

### 3.2 사용 루프 완성도

| 루프 단계 | 완성도 | 배리어 |
|---------|-------|--------|
| Dream 기록 | 95% | 검증 로직 약간 모호 |
| AI 분석 | 90% | 로딩 시간 + 오류 처리 |
| 예보 확인 | 85% | 정보 과다, 신뢰도 미설명 |
| CheckIn | 70% | **시간 목표 미달성** |
| 리포트 | 80% | 정보 과다, 모바일 가독성 |
| **평균** | **84%** | CheckIn 시간 재검토 필요 |

## 4. 정보 아키텍처

### 4.1 네비게이션 구조

```
Router.jsx 분석
├─ Public Routes (인증 전)
│  ├─ /login (이메일/게스트)
│  └─ /signup
├─ Protected Routes (인증 후, 온보딩 필수)
│  ├─ / (Dashboard - 홈)
│  ├─ /onboarding (온보딩 - 완료까지 필수)
│  ├─ /dream (DreamCapture - 기록 + 분석)
│  ├─ /checkin (CheckIn - 멀티 스텝)
│  ├─ /report (WeeklyReport - 분석)
│  ├─ /symbols (SymbolDictionary - 참고)
│  ├─ /search (TimelineSearch - 검색)
│  └─ /settings (Settings - 설정)
└─ BottomNav (탭 6개 + 기본)
   ├─ 홈 (Dashboard)
   ├─ 꿈 기록 (DreamCapture)
   ├─ 체크인 (CheckIn)
   ├─ 리포트 (WeeklyReport)
   ├─ 심볼 (SymbolDictionary)
   └─ 설정 (Settings)
```

### 4.2 깊이 분석

| 페이지 | 깊이 | 경로 |
|--------|------|------|
| Dashboard | 1 | / |
| DreamCapture | 1 | /dream (+ ?id=깟음ID로 상세 뷰) |
| CheckIn | 1 | /checkin |
| WeeklyReport | 1 | /report |
| SymbolDictionary | 1 | /symbols (+ 검색/필터) |
| TimelineSearch | 1 | /search |
| Settings | 1 | /settings |

**깊이 평가**:
- **너무 얕음** (Flat Structure)
- 부작용: 상세 페이지 없음 (꿈 상세, 심볼 상세는 모달/인라인 표시)
- 장점: 빠른 네비게이션, 혼란 적음

### 4.3 페이지 간 관계 (네트워크)

```
Dashboard (중심)
  ├─→ DreamCapture (꿈 기록 즉시 열기)
  ├─→ CheckIn (체크인 빠른 이동)
  ├─→ WeeklyReport (통계 자세히 보기)
  ├─→ SymbolDictionary (최근 심볼 링크)
  ├─→ TimelineSearch (통합 검색 링크)
  └─→ Settings (프로필/설정)

DreamCapture
  ├─← Dashboard (뒤로 가기)
  ├─→ 분석 결과 (인라인 확장)
  ├─→ SymbolDictionary (발견된 심볼 클릭)
  └─→ TimelineSearch (최근 꿈 검색)

SymbolDictionary
  ├─← Dashboard (뒤로 가기)
  ├─→ 심볼 검색 (필터)
  └─→ DreamCapture (관련 꿈 보기 → 구현 안 됨)

Settings
  ├─← Dashboard (뒤로 가기)
  ├─→ 알림 설정 (온보딩과 중복)
  ├─→ Feature Flags (developer 모드)
  └─→ 데이터 export/import
```

**평가**:
- ✅ Dashboard 중심 허브 구조 (명확)
- ⚠️ 일부 링크 구현 안 됨 (심볼 → 관련 꿈)
- ⚠️ 깊은 탐색 경로 제한 (상세 정보 최소화)

### 4.4 정보 흐름 (Content Flow)

```
신규 사용자
  ↓
[Auth] email/password 또는 게스트
  ↓
[Onboarding] 3단계 + 알림/샘플 설정
  ↓
[Dashboard] 빈 상태 (또는 샘플 포함)
  ↓
  ├─ Dream 기록 → AI 분석 (symbol/emotion/theme)
  ├─ 분석 결과 ← SymbolDictionary 연결 (학습)
  ├─ CheckIn 완료 → 예보 생성
  ├─ 예보 검증 (hit/miss 기록)
  └─ 주간 데이터 누적 → WeeklyReport 시각화
     └─ 패턴 인식 → 목표/코치 플랜 제시
        └─ 기록 → 라이프사이클 형성
```

**평가**: 명확하고 논리적 흐름. 데이터 → 분석 → 인사이트 → 행동의 선순환 구조.

#### 접근성 속성 검토

**CheckIn (가장 좋은 예)**:
```jsx
// 라디오 그룹
<div role="radiogroup" aria-label="컨디션 선택">
  <button
    role="radio"
    aria-checked={value === c.value}
    aria-label={`컨디션 ${c.label}`}

  </button>
</div>

// 프로그레스 바
<div

  aria-label={`체크인 진행률 ${Math.round(progress)}%`}

```

**평가**: ✅ 명확한 역할, 레이블, 값 정보

**Dashboard (미흡한 예)**:
```jsx
// aria-label 누락
<button
  onClick={() => navigate('/report')}
  className="flex items-center gap-1 text-sm text-violet-400"
>
  자세히 보기 <ChevronRight className="w-4 h-4" />

```

**평가**: ❌ "자세히 보기"만으로는 목적 불명확

#### 키보드 네비게이션

**테스트 불가** (코드 검토만 가능):
- ✅ Button 컴포넌트: `onClick` 핸들러 지원
- ✅ Link/Navigate: 기본 라우터 사용
- ⚠️ 카드 클릭: `onClick`만 지원, 엔터 키 지원 명시 안 됨
- ⚠️ 토글 칩: `aria-pressed`는 있으나, 포커스 스타일 불명확

**개선 필요**:
```jsx
// 현재
<button onClick={() => handleSelectEmotion(id)}>
  {emotion.name}
</button>

// 개선
<button
  onClick={() => handleSelectEmotion(id)}
  onKeyDown={(e) => e.key === 'Enter' && handleSelectEmotion(id)}
  aria-pressed={selected.includes(id)}
  aria-label={`감정 ${emotion.name}`}
>
  {emotion.name}
</button>
```

#### 색상 대비

**검사 기준**: WCAG AA (최소 4.5:1), AAA (7:1)

**현재 팔레트**:
| 조합 | 대비도 | 등급 |
|------|-------|------|
| 보라색 텍스트 on 검은 배경 | ~6:1 | ✅ AA |
| 연한 회색 텍스트 on 검은 배경 | ~3.5:1 | ❌ 미달 |
| 초록색 텍스트 on 검은 배경 | ~5:1 | ✅ AA |
| 빨강색 텍스트 on 검은 배경 | ~4:1 | ✅ AA |

**문제점**: 연한 회색 (`--text-muted`) 텍스트가 WCAG AA 미달.

**영향 받는 요소**:
- 부제목 ("30초면 끝나요")
- 타임스탬프 ("어제")
- 진행 상황 텍스트 ("아직 기록 없음")

**개선**: `--text-muted` 색상 더 어둡게 조정 (회색 → 중간 회색).

#### 포커스 관리

**검사 결과**:
- ✅ 포커스 순서: 자연스러운 DOM 순서 (수정 필요 없음)
- ❌ 포커스 스타일: 명시적 `:focus` 스타일 부족 (기본 브라우저 스타일에만 의존)

**개선**:
```css
button:focus-visible {
  outline: 2px solid var(--text-primary);
  outline-offset: 2px;
}
```

#### 모바일 접근성

**터치 타겟**:
- ✅ 버튼 최소 크기: 44x44px (지원)
- ✅ 칩/토글: 30-40px (경계선)
- ⚠️ 이모지 버튼: 64x64px (큼, 좋음)

**화면 방향**:
- ✅ SafeArea 컴포넌트로 노치 대응
- ✅ Viewport 설정 명확

#### 종합 접근성 평가

| 항목 | 현황 | 등급 |
|------|------|------|
| 화면 리더 지원 | 부분 지원 (CheckIn ✅, 다른 페이지 ⚠️) | B |

| **Dashboard 카드 수 줄이기** (7→4) | 높음 | 중간 | 1 | 2주 |
| **CheckIn 시간 목표 재조정** (30초 → 60초) | 중간 | 낮음 | 2 | 1주 |
| **토스트 메시지 개선** (진행 상황 표시) | 중간 | 낮음 | 3 | 1주 |
| **WeeklyReport 섹션 축소** | 중간 | 중간 | 4 | 2주 |
| **꿈 기록 후 자동 스크롤** | 낮음 | 낮음 | 5 | 3일 |
| **감정 선택 최대 개수 줄이기** (5→3) | 낮음 | 낮음 | 6 | 1주 |

### 8.2 A/B 테스트 설계

#### 테스트 1: Dashboard 카드 수 줄이기

**가설**: 카드 수를 줄이면 (7→4) 사용자 체류 시간은 감소하지만, 스트레스는 낮아지고 행동 전환율(꿈 기록/체크인)은 올라간다.

**대안**:
- **Control (현재)**: 7개 카드 (예보 + 패턴 + 코치 + 복구 + 리뷰 + 통계 + 꿈 목록)
- **Variant A**: 4개 카드 (예보 + Quick Action + 통계 + 꿈 목록), 기타는 "더 보기"
- **Variant B**: 5개 카드 (예보 + 코치 + 리뷰 + 통계 + 꿈), 패턴/복구 숨김

**측정 메트릭**:
- 체류 시간 (초)
- 스크롤 깊이 (%)
- 클릭률 (꿈 기록 / 체크인)
- 주간 활성 일수

**표본 크기**: 500명 × 2주

**기대 결과**: Variant A 체류 시간 -20%, 꿈 기록 클릭률 +15%

#### 테스트 2: CheckIn 시간 목표 재조정

**가설**: "30초면 끝나요" → "60초면 끝나요"로 변경하면, 사용자는 목표를 현실적으로 지각하고, 완료 후 만족도가 올라간다.

**대안**:
- **Control**: "30초면 끝나요" (불가능함)
- **Variant**: "60초면 끝나요" (달성 가능함)

**측정 메트릭**:
- 평균 완료 시간 (초)
- 완료율 (%)
- 완료 후 만족도 (토스트 확인)
- 주간 체크인 빈도

**표본 크기**: 300명 × 2주

**기대 결과**: 만족도 +25% (목표 달성 감각)

#### 테스트 3: WeeklyReport 정보 아키텍처

**가설**: WeeklyReport를 탭 구조 (Overview / Chart / Insights / Goals)로 분리하면, 정보 과다 느낌이 줄어들고, 각 섹션 클릭률이 올라간다.

**대안**:
- **Control**: 긴 스크롤 페이지 (모든 정보 한 번에)
- **Variant**: 탭 네비게이션 (분리된 보기)

**측정 메트릭**:
- 페이지 체류 시간
- 평균 스크롤 깊이
- 각 섹션 클릭률
- 목표 제시 후 적용률

**표본 크기**: 400명 × 2주

#### 테스트 4: 감정 선택 최대 개수 조정

**가설**: 감정 선택을 5개 → 3개로 줄이면, 체크인 완료 시간이 단축되고, 사용자는 더 중요한 감정에 집중한다.

**대안**:
- **Control**: 최대 5개 ("최대 5개까지 선택할 수 있어요")
- **Variant A**: 최대 3개 ("상위 3개를 선택해주세요")
- **Variant B**: "선택적" 표시 (선택 안 해도 진행 가능)

**측정 메트릭**:
- 체크인 평균 완료 시간
- 감정 선택 개수 평균
- 감정 단계 이탈률
- 주간 체크인 완료율

**표본 크기**: 600명 × 1주 (빠른 테스트)

### 8.3 정성적 피드백 수집 기회

**방법**: In-app 서베이 (체크인 완료 후)

```
"이 체크인이 얼마나 쉬웠나요?"
☆ ☆ ☆ ☆ ☆  (5점 척도)

"가장 어려웠던 부분은?"
□ 꿈 기록
□ 감정 선택
□ 이벤트 선택
□ 시간 입력
□ 없음
```

**기간**: 주 1회 × 4주 (50-100명 응답 목표)

## 9. 주요 발견사항 및 권장사항

### 9.1 핵심 발견사항 요약

#### 강점

✅ **명확한 가치 제안**
- "꿈과 감정의 패턴을 분석하여 더 나은 하루를 예측"
- 사용자가 즉시 이해할 수 있는 핵심 가치
- 온보딩에서 명확히 전달

✅ **따뜻한 톤앤매너**
- 격려적이고 공감적 메시지 일관성
- 사용자를 존중하는 느낌 (예: "조금 더 자세히 적어주세요")
- 브랜드 신뢰도 향상

✅ **견고한 일일 루틴**
- 4-5단계 명확한 루프 (꿈 → 분석 → 예보 → 체크 → 리포트)
- 각 단계에서 의미 있는 피드백 (AI 분석, 예보, 통계)

✅ **접근성 기본 갖춤**
- CheckIn 페이지의 충실한 aria-* 속성
- 터치 타겟 크기 적절
- SafeArea로 디바이스 대응

✅ **게이미피케이션 요소**
- 진행률 바, 연속 기록 스트릭, 목표 체크리스트
- 미묘하지만 효과적인 동기부여

#### 약점

❌ **정보 과다 (Dashboard)**
- 7-8개 카드 동시 표시 → 인지 부하 높음
- 사용자가 어디서 시작해야 할지 불명확
- 스크롤 필수 → 모바일 피로도

❌ **목표와 현실의 갭 (CheckIn)**
- "30초면 끝나요" vs 실제 60-80초
- 실패 경험 → 신뢰도 감소 위험
- 감정/이벤트 단계에서 의사결정 시간 과다

❌ **일부 접근성 미흡**
- aria-label 불균형 (CheckIn ✅, Dashboard ❌)
- 회색 텍스트 색상 대비 미달 (WCAG AA)
- 명시적 포커스 스타일 부족

❌ **WebKit 기능 의존성**
- 음성 녹음 (Web Speech API) - Safari 지원 제한
- Capacitor 플러그인 - 웹에서는 동작 안 함
- 점진적 개선(graceful degradation) 부족

❌ **데이터 수집 초기 단계**
- 1주일 미만 데이터로는 패턴 감지 불가 (현재 3+일 필요)
- 빈 상태(empty state)에서 사용자 불안감 가능

#### 중립 발견사항

⚠️ **피쳐 플래그 구조**
- healthkit, UHS, saju, b2b 4개 기능 조건부
- 장점: 점진적 출시, A/B 테스트 용이
- 단점: 사용자 경험 불일치, 설명 필요

⚠️ **온보딩 샘플 데이터**
- 샘플 데이터 로드 상태 추적 안 됨 → 중복 로드 가능
- 사용자가 실제 데이터인지 샘플인지 구분 불명확

### 9.2 우선순위 권장사항

#### Phase 1 (1-2주) - 즉시 개선

**1. Dashboard 카드 정렬**
- [ ] 핵심 카드 3개만 위쪽 (예보 + Quick Action + 통계)
- [ ] 보조 카드는 "더 자세히" 섹션으로 (코치/복구/리뷰 축소)
- [ ] 컬러 일관성 (보라색 중심)
- 예상 영향: 체류 시간 -15%, 스트레스 -20%

**2. CheckIn 시간 목표 재조정**
- [ ] "30초면 끝나요" → "1분이면 충분해요" 변경
- [ ] 또는 실제 완료 시간 측정 후 정직한 표시
- 예상 영향: 완료 후 만족도 +25%

**3. 접근성 기본**
- [ ] `--text-muted` 색상 더 어둡게 (회색 → 중간 회색)
- [ ] Dashboard 주요 버튼에 aria-label 추가
- [ ] `:focus-visible` 스타일 추가
- 예상 영향: 접근성 등급 C+ → B

#### Phase 2 (2-4주) - 사용자 피드백 기반

**4. WeeklyReport 재구조화**
- [ ] 탭 네비게이션 추가 (Overview / Insights / Goals)
- [ ] 각 탭 50% 정보 축소
- 예상 영향: 체류 시간 안정화, 목표 적용률 +20%

**5. In-app 피드백 수집**
- [ ] 체크인 완료 후 NPS/CSAT 서베이
- [ ] 주 1회 × 4주 (50-100명 응답)
- [ ] 정성적 피드백 (어려운 단계, 불명확한 기능)
- 예상 영향: 다음 개선 우선순위 데이터 확보

**6. A/B 테스트 실행**
- [ ] Test 1: Dashboard 카드 수 (500명 × 2주)
- [ ] Test 2: CheckIn 시간 목표 (300명 × 1주)
- 예상 영향: 데이터 기반 의사결정

#### Phase 3 (1개월) - 기능 확장

**7. 온보딩 개선**
- [ ] 샘플 데이터 로드 상태 Zustand에 저장
- [ ] 알림 권유 3단계 (지금/나중/거절)
- 예상 영향: 재진입율 +10%

**8. 감정 선택 UX**
- [ ] 최대 3개로 줄이기 또는 "상위 3개만 선택" 표시
- [ ] 음성 비고: "가장 강한 감정 3개를 선택해주세요"
- 예상 영향: CheckIn 완료 시간 -20%

**9. 게이미피케이션 확장**
- [ ] 성취 배지 시스템 (7일 연속 체크인 등)
- [ ] 주간 목표 달성 축하 모달
- 예상 영향: 주간 활성 일수 +15%

### 9.3 장기 비전 (3-6개월)

- 사용자 커뮤니티 (심볼 공유, 꿈 일기 비공개 연동)
- AI 개인화 (사용자별 심볼 의미학 학습)

## 부록 C: 용어 정의

- **마찰도 (Friction)**: 사용자가 의도한 행동을 완료하기 위해 필요한 노력 정도
- **드롭오프**: 사용자가 사용 흐름을 중단하는 지점
- **루프 (Loop)**: 반복되는 사용 패턴 (꿈 기록 → 체크인 → 예보)
- **인지 부하**: 화면에 표시된 정보량이 사용자 뇌에 주는 부담
- **감정 곡선**: 사용자 여정 전체에서 경험하는 감정의 변화
- **톤앤매너**: 앱이 사용자와 커뮤니케이션하는 방식/태도
- **A/B 테스트**: 두 가지 대안을 임의로 할당하여 효과 측정하는 실험
- **WCAG**: Web Content Accessibility Guidelines (웹 접근성 표준)

## 결론

DreamSync는 **명확한 가치 제안, 따뜻한 톤, 견고한 핵심 루프**를 갖춘 UX를 보유하고 있습니다.

하지만 **정보 과다(Dashboard), 시간 목표 불일치(CheckIn), 부분적 접근성 미흡**이 사용자 경험을 저해하고 있습니다.

**즉시 개선** (1-2주) 을 통해 Dashboard 간결화, CheckIn 시간 재조정, 접근성 강화를 하면, **사용자 만족도 +20-30%**, **이탈률 -10-15%** 개선을 기대할 수 있습니다.

이후 **사용자 피드백 수집** 및 **A/B 테스트**를 통해 데이터 기반 의사결정으로 점진적 개선을 추진하기를 권장합니다.
